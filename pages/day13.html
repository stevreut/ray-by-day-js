Day 13 - class Vector3D

In mathematics, a vector can be thought of as an arrow that proceeds in some direction for some length. If two vectors have the same direction and have the same length then they are considered equal, but only if both direction and length are equal, or if both have zero length in which case direction is meaningless. Vector math provides for a number of operations and functions that return vectors or numbers. (Since vector math often deals with vectors in combination with simple real numbers, real numbers are referred to as "scalars" when used in the context of vector math.)

The power and utility of vector math is that vectors, vector operations, and vector functions generally have both a numeric and a geometric aspect. That being the case, vectors provide an elegant way for modelling the geometric notions of position and direction in a way that lends itself to pure numeric representation and calculation. It is this marriage of geometry and pure numeric calculation that makes vectors so useful in the context of ray tracing graphics.

I won't try to give a full lesson on vectors in this site. However, having at least a basic familiarity with vectors is probably a good idea before diving into how they are used in graphics and, specifically, in ray-tracing. Following are a few resources that might be useful but there are many others as well so I encourage either using these resources or finding some others that suit you.

Some vector resources

class Vector3D

For the purposes of this site, we need a JavaScript class for representing the attributes and behaviors of vectors. Specficially, since we will be modelling a virtual three-dimensional environment, we are implementing a class specifically tailored to modelling vectors in a three-dimensional space. Note that vectors can be defined to have any degree of dimensionality and, regardless of the number of dimensions, all vectors have certain shared characteristics and operations ("methods" for our purposes). Arguably, a more disciplined implementation would have have had us implement a more generic class Vector first and then extend that to a more specialized 3D vector class, but I've chosen to skip that step and have gone straight to implementing class Vector3D without implementing a super class. See the vector3d.js link in the Resources used in this page section at the bottom page to see the full implementation.

Essentially, a Vector3D object is an encapsulation of a set of three real (floating point) numbers. The Vector3D objects are immutable, meaning that the defining attributes of a Vector3D object (its x, y, and z coordinates) cannot be changed once that object is created/instantiated. This is done to minimize the amount of unnecessary repeated calculation.

An example: the magn() method returns the "magnitude" of the vector (it's length). Calculating the magnitude requires three multiplications, two additions, and a square root, and every one of those operations eats up a little CPU time - especially the square root. In this implementation, the private attribute #magnVal is initially set to null. When magn() is called, if #magnVal is null then the full magnitude calculation is executed but, before returning the resulting value, #magnVal is assigned that value. So next time magn() is called for that particular vector object, the method will detect that #magnVal is non-null and will simply return that value rather than recalculate it. Because the Vector3D object is immutable, it can be reliably assumed that the magnitude also has not changed. Now any code using the Vector3D class can call magn() knowing that there will be no unnecessary redundant calcuation of the vector's magnitude.

Managing the amount of calculation is key in ray-tracing and eliminating redundant or unnecessary calculations becomes increasingly important as the virtual environments we create become more complex and the resolution of the images increases.

Another example of built-in elimination of unnecessary calculation is illustrated by the inclusion of the magnSqr() method. This method returns the square of the magnitude. This is not really a standard vector function, but it comes up often enough to justify a special method for returning this value. It turns out that the square of the magnitude is less computationally expensive than the magnitude is. This is because the magnitude is the square root of the sum of the squares of the components and, consequently, the square of the magnitude is the square of that square root - i.e. the end result of the same calculation but without the final square root applied. We could have defined magnSqr() as returning the square the result from magn() but that would have required considerably more CPU time. Also, as with magn(), we rely on the immutability of Vector3D objects so that we never calculate the value returned by magnSqr() more than once. (The relationship between magn() and magnSqr() and related private variables actually goes a bit deeper, so feel free to look at the code.)
